# Vibe Coding 的崩潰：一位開發者七個月 AI 輔助開發的深刻反思

**資料來源**：blog.k10s.dev
**原文連結**：https://blog.k10s.dev/im-going-back-to-writing-code-by-hand/
**文章發布時間**：2026-05-09
**市場訊息發布時間**：2026-05-09
**Hacker News 熱度**：254 點，105 條留言

## 緣起：一個 AI 驅動的實驗

一位開發者在 2025 年 9 月開始了一個野心勃勃的實驗：完全依賴 Claude 等 AI 工具來開發一個 GPU 感知的 Kubernetes 儀表板命名為 k10s。這項實驗的目標是探究「在不直接參與程式碼撰寫的情況下，能夠開發出多大規模的軟體」。經過 7 個月、234 次提交、約 30 個週末的努力，這個專案最終宣告失敗，作者決定將整個程式碼庫歸檔，並從零開始用 Rust 重寫。

作者在 Hacker News 上發表此文後引起社群高度關注，不僅因為這是一個失敗案例的誠實分享，更重要的是它揭示了 AI 輔助開發中許多被成功案例掩蓋的深層問題，這些問題只有在專案規模擴大到一定程度後才會顯現。

## 經驗總結：五條關鍵教訓

### 教訓一：AI 撰寫功能，不撰寫架構

作者最初的體驗是令人興奮的：首批幾週，每次要求 Claude 添加功能都能立即實現。資源列表視圖、命名空間過濾、日誌串流、描述面板、鍵盤導航，所有功能都能在單次會話中完成。作者形容當時的速度是平常的十倍，感覺極佳。

然而，問題隨之而來。每個功能都是在「讓它現在就能動」的語境下實現，完全沒有考慮到其他 49 個功能共享同一個狀態。當 GPU 機群視圖這個主要功能被添加時，AI 在第一次嘗試就生成了正確的代碼，但切換回 Pods 視圖時，什麼都沒有顯示、即時更新停止了、切換到節點視圖顯示的是機群視圖的過濾數據。「神物件（God Object）已經吞噬了自己」。

這揭示了 AI 的根本限制：它能優化完成每個具體提示，但無法維護全盤架構的一致性。人工架構設計是不可替代的。

為了防止這個問題，作者建議在 `CLAUDE.md` 中明確定義架構不變量，例如：每個視圖都實現 View trait，視圖不能直接訪問其他視圖的狀態，所有非同步數據都通過 AppMsg 變體到達，不允許對後台任務直接修改字段，添加新視圖無需修改現有視圖，App 結構是一個薄路由器，只擁有導航和消息分發，不擁有其他內容。

### 教訓二：神物件是 AI 的預設產物

作者發現 AI 傾向於生成「單一結構承載所有內容」的代碼，因為這滿足了即時提示的最小化要求。但這導致了嚴重的問題。在 1690 行的 `model.go` 文件中，一個名為 `Model` 的結構包含了所有內容：UI 組件、K8s 客戶端、日誌、描述、機群、鼠標處理、導航歷史，所有東西都塞在一個結構中。而 `Update()` 方法是一個 500 行的函數，通過 `msg.(type)` 進行 110 個 switch/case 分支。

更糟的是，由於沒有視圖隔離，鍵盤處理變成了噩夢。在該文件的 1690 行代碼中，有 9 個手動 `= nil` 清理行分散在各處。少一個，就會得到來自先前視圖的「幽靈數據」。這就是沒有視圖隔離的代價。AI 看不到這個模式隨時間衰敗，因為每個提示只觸及一個代碼路徑。

作者建議在 `CLAUDE.md` 中設定狀態所有權規則：不要為視圖特定狀態向 App/Model 結構添加字段。每個視圖都是實現 View trait/interface 的獨立結構。每個視圖聲明自己的鍵盤綁定。App 將鍵盤分發到活動視圖。如果要添加鍵盤綁定，請將其添加到相關視圖的鍵盤映射中。添加視圖意味著添加一個文件，如果你的更改需要修改現有視圖，請停止並詢問。

另外關於結構設定也變得複雜，鍵盤綁定是 `s`，在日誌視圖中是「自動滾動」，在 pods 視圖中是「shell」，在容器視圖中是「進入容器」，這都通過 `m.currentGVR.Resource` 進行判斷。

作者將此比喻為就像文學中的「破折號運用過度」，對於 AI 編碼來說，「神物件」也會導致代碼變得混亂和難以維護。有點像品味問題，多了就變得廉價感。

還有像是按下 Enter 選擇功能的邏輯，得用一連串判斷式來處理不同視圖下的行為，在上下文 contexts 巢狀判斷，然後在命名空間視圖又是另一串邏輯，日誌視圖則是直接返回。

每個視圖都是扁平分發中的條件，該文件中有 20 多次 `m.currentGVR.Resource ==` 作為類型區分符的出現。不是類型。而是字串比較。每個新視圖意味著觸及每個處理程序。

### 教訓三：速度假象擴大了範圍

這是心理層面，而非技術層面，作者認為這是最危險的。

當作者開始 k10s 時，目標是建一個 GPU 專屬工具，為那些運行訓練集群的人，一個作者身為社群成員的利基受眾。但 vibe-coding 讓一切感覺便宜。

「噢，我可以在一個會話中添加 pods 視圖？讓我也添加 deployments 吧。還有 services。還有完整的指令面板。還有滑鼠支援。還有 contexts。還有 namespaces。」突然間，作者在建 k9s，一個通用的 Kubernetes TUI，為每個人。因為 AI 讓每個功能感覺免費。

但這不會免費，每個功能都是神物件中的另一個分支。速度感讓你以為你在贏，直到一切同時崩潰。作者建議寫一份願景文件，明確說明你不想為誰建，並把範圍邊界放在你的 `CLAUDE.md` 中。

CoffeeScript 和 Elm 的發明者 Jeremy Ashkenas 曾說過類似的話，創造性工作最重要的是知道什麼不該做，而非做了多少。

作者也提到 scope 定義可以寫在文件裡「範圍（不要擴展超出此範圍）」、k10s 是給 GPU 集群運營商的，不是所有 Kubernetes 用戶。支援的視圖：fleet、node-detail、gpu-detail、workload，就這樣。不要添加通用資源視圖（pods、deployments、services）。不要添加複製 k9s 功能的功能。如果某個功能需求不為運行 GPU 訓練作業的人服務，則拒絕它。

### 教訓四：位置數據是定時炸彈

k10s 中的每個資源都從 Kubernetes API 獲取並立即扁平化，欄位識別完全取決於位置。例如排序函數中，`ra[3]` 是 Alloc，`ra[2]` 是 Compute，`ra[0]` 是 Name，這些是魔法數字，唯一連接索引 3 和「Alloc」的是註釋以及在 `resource.views.json` 中定義的欄位順序。

如果在 Instance 和 Compute 之間新增一欄呢？那麼負責排序的所有條件渲染、每次說 `ra[2]` 或 `ra[3]` 的地方現在都會沈默地出錯。編譯器也幫不上忙，因為都是 `[]string`。JSON 配置也無法表達排序行為、條件渲染或自定義鑽取目標，所以那些存在於 Go 代碼中，硬編碼了 JSON 的位置假設。

AI 生成此模式是因為它是從「獲取數據」到「渲染表格」的最短路徑，一個 `[]string` 立即滿足任何表格小部件。但在生成的 6 個月後，你會調試為什麼排序會將「Name」值放入「Alloc」欄目中，但如果你使用型態結構作為索引參數就可以避免掉這種問題，如果不使用型態化結構就很難使用範型來輕鬆地手動處理字串轉換或過濾等複雜業務邏輯，大部分事情都硬編寫在加工邏輯裡，這會讓之後的新增欄位之後需要同步修改多個功能才能將欄位正確地渲染到表格上。

作者建議在 `CLAUDE.md` 中加入資料表示相關指令「不要將結構化資料扁平化成 []string、Vec<String> 或位置陣列。所有資料都作為型態化結構流動，直到 render()/view() 呼叫為止。欄位識別來自結構欄位名稱，而非陣列索引。排序函數對型態化欄位操作，而非像 row[3] 的位置存取。字串僅在 render()/view() 函數內部建立以供顯示」。

從而你的型態結構讓不可能的狀態變得不可能。作者舉了個例子，你無法在欄目是具名欄位時按錯誤欄目排序，你無法意外地将分配字串與名稱進行比較，編譯器會為你強制執行此操作，而 AI 將始終選擇 `Vec<String>`，因為它能更快滿足提示。

### 教訓五：AI 不擁有狀態轉換

Bubble Tea 架構有一個美妙的思想：`Update()` 是狀態改變的唯一地方，由消息驅動，但 k10s 違反了這一點，`updateTableMsg` 處理程序生成了在 goroutine 中修改結構字段的方法。

返回的函數被 Bubble Tea 在單獨的 goroutine 中執行，它會讀取和寫入資源表、視圖寬度、表格字段。同時，`View()` 在主 goroutine 上讀取相同的字段，沒有鎖。沒有 Mutex。通道 `<-m.updateTableChan` 會阻塞 goroutine，直到某人發送更新訊號，但沒有任何東西阻止 `View()` 在狀態寫入到一半時進行讀取。

這是教科書級別的數據競爭，它 99% 的時間都有效，會以讓你以為自己瘋了的方式損壞顯示 1% 的時間。AI 生成這個是因為「而在閉包中直接改變它」是工作代碼的最短路徑。適當的訊息傳遞將訊息發送回 `Update()`，讓 `Update()` 在主循環上應用更改。這需要更多的類型、更多的管道。

作者經歷到這種數據競爭導致了畫面顯示錯亂的情況，這種產生錯誤的除錯過程非常消耗寶貴的時間，很難確認問題所在的根本原因，也無法穩定的再現問題，因為數據競爭取決於並發環境的時機與及其他變數導致的。

作者建議所有對渲染可見狀態的改變都發生在主循環上。後台工作產生數據。他們將其作為訊息發送，主循環接收訊息並應用它。這是線上 UI 代碼中不能打破的唯一規則。

用戶可以自己在筆記型電腦上運行大規模 AI 模型讓人心動，能做到本地運行的最重要的原因是不需要連網就能使用，可以在飛機上用，然後成本只考量電費，不用付訂閱費，不過環境衝擊最大的可能還是在訓練階段，但就有機會少用一些數據中心。

## 重新思考 AI 在軟體開發中的角色

這篇文章在 Hacker News 上獲得廣泛討論，許多開發者分享了類似的經驗。核心問題在於：AI 是一個強大的工具，但不是一個獨立的架構師。

文章發布時間是 2026-05-09，正值 AI 輔助開發工具如 GitHub Copilot、Cursor、Claude Code 等普及的高峰期。開發者們在嘗試「vibe-coding」——即完全依賴 AI 輸入代碼，自己只提供自然語言指令——這種模式看似能極大幅提升生產力，但本文作者用七個月的實驗證明了這種方法在高規模專案中的不可行性。

重要的是要理解，文章作者並不是完全反對 AI 輔助開發，而是在反對「完全替代人工思考」的使用方式。他決定從零重寫 k10s，但這次他會：
1. 在任何代碼編寫之前，自己完成設計工作
2. 提供具體的介面、訊息類型、所有權規則
3. 將架構決策以書面形式記錄，在第一個提示之前就能做出

作者在文章結尾問道：「我正在用 Rust 重寫 k10s。不是因為 Rust 更好，而是因為它是可以駕馭的語言。我已經寫了足夠多的 Rust，感覺有些東西錯了，在我能解釋原因之前。那種直覺是 vibe-coding 無法取代的東西。」

還有一個更簡單的想法：作者慣用 Rust，在需要架構與抽象的時候，偏向函數式語言開發者通常會很堅持和固執設計架構時要非常謹慎和明確。設計工作中的都是功能跟程式碼可以做到非常強型別跟不可變狀態絕對安靜地 typesafe，清楚的介面是保持umutability 和一階狀態流動的關鍵。

作者總結，這篇文章的價值不在於反對 AI，而在於提醒開發者：AI 是放大鏡，它會放大你的設計優點，也會放大你的設計缺陷。一個好的開發者應該知道什麼時候讓 AI 幫助，什麼時候自己握緊方向盤。
